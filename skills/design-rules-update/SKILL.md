# Design Rules Update

デザイン作業で得た知見を `~/.claude/design-rules.md` に蓄積し、プロジェクト横断で再利用可能なデザインルール集を育てるスキル。

## いつ実行するか

以下のいずれかに該当したら、このスキルを実行する:

1. **ユーザーが明示的にルール追加を依頼した** — 「このルール覚えて」「design-rulesに追加して」など。この場合は常に起動する
2. **再利用可能な視覚設計の原則が得られた** — デザイン作業の結果、他プロジェクトでも適用できる原則が新たに生まれた場合に起動する

### 起動しないケース

以下のような変更では起動しない。ノイズを防ぐために重要:

- テキスト・文言の差し替えのみ
- 余白やサイズの微調整（既存ルールに沿った調整）
- プロジェクト固有のブランドカラーやブレークポイント値の設定
- バグ修正に伴うスタイル修正（意図的なデザイン判断ではないもの）
- デバッグ用の一時的なスタイル変更

迷ったら起動しない。ルールの蓄積は量より質を重視する。

## 実行手順

### Step 1: 変更内容の把握

今回のセッションでデザインに関連する変更を特定する。以下のソースを使う:

- `git diff` や `git log` で直近の変更差分を確認
- セッション中に行ったファイル編集の内容（対象: `*.css`, `*.scss`, `*.tsx`, `*.jsx`, `tailwind.config.*`, デザイントークン定義ファイル, UIコンポーネント等）
- ユーザーからの明示的な指示

### Step 2: ルールの抽出

変更内容から、今後のデザイン作業で再利用できる知見を抽出する。

#### 抽出の基準

以下の全てを満たすもののみ抽出する:

- **今回の変更から実証的に得られた知見か** — 一般的なデザイン論や推測ではなく、実際の変更・ユーザー指示から直接導けること。AI が「ついでに」一般論を足してはいけない
- **汎用性があるか** — このプロジェクト固有ではなく、他のプロジェクトでも適用できる原則であること
- **具体的か** — 「見た目を良くする」のような曖昧なものではなく、実行可能な粒度であること
- **将来の意思決定に使えるか** — 単発の解決策ではなく、今後同じ判断を迫られたときの指針になること

#### ルールの形式

```
- [具体的なルール] — [理由]
```

例外条件がある場合は添える:
```
- [具体的なルール] — [理由]（例外: [条件]）
```

例:
```
- ボタンのホバー状態は opacity ではなく背景色の変化で表現する — opacity だと中のテキストも薄くなり視認性が下がるため（例外: アイコンのみのボタンは opacity でも可）
```

### Step 3: 既存ルールとの照合

`~/.claude/design-rules.md` を読み、以下を判定する:

- **重複** — 表現が違っても意味が同じなら重複とみなす。追加しない
- **近似** — 似ているが粒度や範囲が異なる場合は、既存ルールと統合して1つにまとめることを優先する。新規追加より統合を選ぶ
- **矛盾** — 既存ルールと矛盾する場合は、ユーザーに「既存ルール A と矛盾しますが、更新しますか？」と確認してから対応する。勝手に上書きしない
- **新規** — 既存のどれとも重ならない場合のみ新規追加する

### Step 4: design-rules.md への追記

#### ファイルが存在しない or 空の場合

新規作成して以下のヘッダを書き、そのうえでルールを追記する:

```markdown
# デザインルール
```

カテゴリ見出しは必要になった時点で作る。空のカテゴリを予め並べない。

#### 追記のフォーマット

```markdown
# デザインルール

## [カテゴリ名]
- [ルール] — [理由]
- [ルール] — [理由]（例外: [条件]）
```

カテゴリの例: カラー・配色 / タイポグラフィ / レイアウト・スペーシング / コンポーネント / UI要素のボーダー / アニメーション・トランジション / レスポンシブ / アクセシビリティ

カテゴリは固定ではない。既存のカテゴリに該当する場合はそこに追記し、どれにも当てはまらない場合は適切な名前で新設する。

### Step 5: ユーザーへの報告

結果に応じて以下のいずれかの形式で報告する:

**追記した場合:**
```
design-rules.md に追記しました:
- [カテゴリ名]: [追記したルールの要約]
```

**統合した場合:**
```
design-rules.md の既存ルールを更新しました:
- [カテゴリ名]: [変更内容の要約]
```

**追加なしの場合:**
```
今回の変更からは新たなデザインルールの追加はありません。
（理由: [既存ルールでカバー済み / プロジェクト固有の変更のため / etc.]）
```

**確認が必要な場合:**
```
既存ルールとの矛盾があります:
- 既存: [既存ルール]
- 今回: [新しいルール]
更新しますか？
```

## 注意事項

- 1回の追記で大量のルールを追加しない。本当に再利用価値のあるものに絞る（目安: 1〜3個）
- プロジェクト固有の設定値（特定のブレークポイント値、特定のブランドカラーのHEX値など）はルールに含めない
- ユーザーが「追加しないで」と言った内容は追加しない
- ルールの削除・修正をユーザーが求めた場合は対応する
- ルールが増えてきたら、同カテゴリ内で関連するルールが近くに並ぶよう整理する
